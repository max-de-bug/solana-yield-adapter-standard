title: Registry

The **Adapter Registry** is the trust anchor for the ecosystem. Only adapters with `Approved` status can be routed through the [dispatcher](/dispatcher). The registry supports a two-tier authority model: a **governance authority** (full control) and an optional **guardian** (day-to-day approval/revocation).

## Lifecycle

```mermaid
stateDiagram-v2
    [*] --> Proposed: propose_adapter()
    Proposed --> Approved: approve_adapter()
    Approved --> Revoked: revoke_adapter()
    Revoked --> Approved: approve_adapter()
    Revoked --> [*]
```

| Status | Dispatcher routing | Who can set |
|---|---|---|---|
| `Proposed` | Blocked | Anyone via `propose_adapter` |
| `Approved` | Allowed | Authority or guardian via `approve_adapter` |
| `Revoked` | Blocked | Authority or guardian via `revoke_adapter` (`approve_adapter` also accepted — re-approve from `Revoked`) |

## Instructions

### `initialize`

Creates `RegistryState` with the deployer as initial governance authority.

### `propose_adapter`

**Permissionless.** Registers a new adapter candidate.

<ParamField path="name" type="string" required>
  Human-readable name, max 32 characters.
</ParamField>

<ParamField path="metadata_uri" type="string" required>
  Off-chain metadata URI (JSON), max 200 characters.
</ParamField>

Also requires: `adapter_program_id`, `underlying_mint`.

### `approve_adapter`

**Authority or guardian.** Transitions `Proposed` → `Approved` or `Revoked` → `Approved` (re-approval after revoke). Sets `approved_at` timestamp.

### `revoke_adapter`

**Authority or guardian.** Transitions `Approved` → `Revoked`. Existing positions are not auto-closed — integrators should surface revocation to users.

### `set_guardian`

**Authority only.** Sets or clears the optional guardian role. Pass `Pubkey::default()` to remove the guardian.

The guardian can approve and revoke adapters but cannot set a new guardian or transfer governance.

### `nominate_governance`

**Authority only.** Sets the `pending_authority` field. The current authority nominates a successor; governance is not transferred until the nominee accepts.

### `accept_governance`

**Pending authority only.** Finalizes the transfer: upgrades `pending_authority` to `authority` and clears the nomination. This two-step pattern prevents accidental loss of control — if a nomination is sent to the wrong address, the current authority can overwrite it before acceptance.

### `force_transfer_governance` (dev/test only)

**Hardcoded admin key only.** Immediately transfers governance to a new authority without the two-step nomination process. Exists solely as an escape hatch for Surfpool persistent state where governance was transferred to a throwaway keypair in a prior run. The admin key is `Pubkey::default()` in production builds.

## `RegistryState` account

| Field | Type | Description |
|---|---|---|
| `authority` | `Pubkey` | Governance authority (transfer governance, set guardian) |
| `guardian` | `Option<Pubkey>` | Optional guardian (approve/revoke adapters) |
| `pending_authority` | `Option<Pubkey>` | Nominated successor for two-step transfer |
| `total_proposed` | `u64` | Cumulative proposals |
| `total_approved` | `u64` | Current approved count |

## `AdapterEntry` account

| Field | Type | Description |
|---|---|---|
| `adapter_program_id` | `Pubkey` | Adapter program to route to |
| `name` | `String[32]` | Display name |
| `status` | `enum` | `Proposed` / `Approved` / `Revoked` |
| `underlying_mint` | `Pubkey` | Token this adapter accepts |
| `metadata_uri` | `String[200]` | Off-chain metadata |
| `vault_state_seed` | `Vec<u8>[32]` | PDA seed bytes for the adapter's vault state account |
| `vault_authority_seed` | `Vec<u8>[32]` | PDA seed bytes for the adapter's vault authority signer |
| `proposer` | `Pubkey` | Original proposer |
| `proposed_at` | `i64` | Unix timestamp |
| `approved_at` | `i64` | Approval timestamp (0 if never approved) |
| `revoked_at` | `i64` | Revocation timestamp (0 if never revoked) |

## Example: register an adapter

```typescript
// Propose with the PDA seeds the adapter uses for its vault state and vault authority
await registryProgram.methods
  .proposeAdapter(
    "My Adapter",
    "https://example.com/adapter.json",
    "my_vault_state",
    "my_vault_authority",
  )
  .accounts({
    proposer: wallet.publicKey,
    registryState,
    adapterEntry: adapterEntryPda(adapterProgramId),
    adapterProgram: adapterProgramId,
    underlyingMint: USDC_MINT,
    systemProgram: SystemProgram.programId,
  })
  .rpc();

// Authority or guardian approves
await registryProgram.methods
  .approveAdapter()
  .accounts({ authority: wallet.publicKey, registryState, adapterEntry })
  .rpc();
```

## Post-deployment checklist

<Steps>
  <Step title="Initialize registry">
    `initialize()` with deployer as governance.
  </Step>
  <Step title="Propose each adapter">
    One `AdapterEntry` PDA per adapter program ID. Each proposal includes the vault state seed string.
  </Step>
  <Step title="Approve adapters">
    Governance calls `approve_adapter` for each reference adapter.
  </Step>
  <Step title="Nominate new governance">
    `nominate_governance` to set multisig as pending authority.
  </Step>
  <Step title="Accept governance">
    Multisig calls `accept_governance` to finalize the transfer.
  </Step>
</Steps>

See [Deployment guide](/guides/deployment) for the full devnet workflow.
