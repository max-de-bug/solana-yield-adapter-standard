## Overview

Drift's Insurance Fund unstaking has a protocol-enforced cooldown. This adapter uses a **two-phase withdrawal model** that directly maps to Drift's lifecycle:

1. **`withdraw` (request phase)** ŌĆö locks shares, records slippage floor, creates a `WithdrawalTicket`
2. **`settle_withdrawal` (execution phase)** ŌĆö after cooldown, runs CPI and transfers underlying

## Two-phase withdrawal

```rust
pub const UNSTAKE_COOLDOWN_SECONDS: i64 = 13 * 24 * 60 * 60; // 1,123,200s
```

### Phase 1 ŌĆö `withdraw` (request)

Shares are locked (deducted from `Position.shares`) and a `WithdrawalTicket` is created with `unlock_ts = now + cooldown`. Slippage is checked against the pool ratio **at request time**, not 13 days later. No CPI call or token transfer occurs.

### Phase 2 ŌĆö `settle_withdrawal`

After `unlock_ts` has passed, the user calls `settle_withdrawal`. The adapter:
1. Verifies `clock.unix_timestamp >= ticket.unlock_ts`
2. Computes underlying at the current pool ratio (not the request-time ratio ŌĆö the user accepted this when requesting)
3. Transfers underlying from vault to user
4. Emits `WithdrawEvent`
5. Closes the ticket account

### `cancel_unstake`

During the cooldown window, the user may cancel the unstake request. Locked shares are returned to `Position.shares` and the ticket is closed.

## Instructions

| Instruction | Who can call | Description |
|---|---|---|
| `initialize` | Authority | Creates vault state |
| `deposit` | Anyone | Deposit underlying, mint shares |
| `withdraw` | Anyone | **Request phase** ŌĆö locks shares, creates ticket |
| `settle_withdrawal` | Position owner | **Execution phase** ŌĆö after cooldown, claim underlying |
| `cancel_unstake` | Position owner | Cancel pending unstake during cooldown |
| `current_value` | Anyone | Query position value in underlying units |
| `toggle_status` | Vault authority | Active Ōåö DepositsPaused Ōåö Paused |
| `set_unstake_cooldown` | Vault authority | Configure cooldown duration (seconds) |

## Error codes

| Code | Name | Description |
|---|---|---|
| `7000` | `CooldownNotElapsed` | Settlement called before `unlock_ts` |
| `7001` | `TicketNotSettled` | Withdrawal ticket has not been settled |
| `7002` | `TicketAlreadySettled` | Ticket has already been settled |

## Cooldown configuration

Each vault stores `unstake_cooldown_seconds` (default: `1_123_200` = 13 days). The authority can change this per-vault via `set_unstake_cooldown`. For testing, set to `0` for instant settlement.
