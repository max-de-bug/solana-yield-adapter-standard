# Build your own adapter

<Info>
**Goal:** An experienced Solana/Anchor developer can ship a **conformant reference adapter** in **under one day** by copying patterns from this repository—not by integrating a live protocol on day one.
</Info>

## Time budget

| Phase | Duration | Output |
|-------|----------|--------|
| Scaffold + trait dependency | 15 min | Compiling Anchor workspace |
| Vault + position state | 30 min | `MyVaultState` + `AdapterPosition` PDAs |
| `deposit` / `withdraw` / `current_value` | 2–3 h | Three instructions + events |
| Tests (local) | 1–2 h | deposit → value → withdraw |
| Registry proposal | 10 min | On-chain `propose_adapter` |
| **Optional:** protocol CPI | 1–2+ weeks | Production integration |

## Prerequisites

- Rust 1.75+, Solana CLI **2.2.20**, Anchor CLI **1.0.1**
- Node.js 18+ for TypeScript tests
- Read [Adapter Standard](/adapter-standard) and skim [`adapter-kamino`](https://github.com/max-de-bug/solana-yield-adapter-standard/tree/main/programs/adapter-kamino) (golden reference)

<Steps>
  <Step title="1. Scaffold (15 min)">
    ```bash
    anchor init my-yield-adapter
    cd my-yield-adapter
    ```

    Add the trait crate (path or git after you publish this repo):

    ```toml
    # programs/my-yield-adapter/Cargo.toml
    [dependencies]
    anchor-lang = "1.0.1"
    anchor-spl = "1.0.1"
    yield-adapter-trait = { path = "../crates/yield-adapter-trait" }
    ```

    ```bash
    npm install @anchor-lang/core@1.0.1
    ```
  </Step>

  <Step title="2. Vault state (15 min)">
    Create `programs/my-yield-adapter/src/state.rs`:

    ```rust
    use anchor_lang::prelude::*;

    yield_adapter_trait::define_adapter_position!();

    #[account]
    #[derive(Debug, InitSpace)]
    pub struct MyVaultState {
        pub authority: Pubkey,
        pub underlying_mint: Pubkey,
        pub total_underlying: u64,
        pub total_shares: u64,
        pub status: VaultStatus,
        pub bump: u8,
    }

    pub const VAULT_STATE_SEED: &[u8] = b"my_vault_state";
    pub const VAULT_AUTHORITY_SEED: &[u8] = b"my_vault_authority";
    ```

    <Warning>
    Use **unique** PDA seed prefixes (`my_vault_state`, not `kamino_vault_state`). The `define_adapter_position!` macro must expand **inside your adapter program** so Anchor assigns the correct program owner to position accounts.
    </Warning>

    Use `VaultStatus` from `yield-adapter-trait` for the `status` field — store the enum directly (`VaultStatus::Active`), not a raw `u8`.
  </Step>

  <Step title="3. Deposit (45 min)">
    Required accounts: `user`, `vault_state`, `user_position` (init_if_needed), `user_token_account`, `vault_token_account`, `token_program`, `system_program`.

    ```rust
    use yield_adapter_trait::{
        shares_for_deposit, DepositEvent, YieldAdapterError, ADAPTER_POSITION_SEED,
    };

    // In handler:
    let shares = shares_for_deposit(
        amount,
        vault.total_underlying,
        vault.total_shares,
    )?;

    // Slippage protection: check minimum output before any state changes.
    require!(shares >= min_shares_out, YieldAdapterError::SlippageExceeded);

    // SPL transfer user → vault, then update vault totals and position.receipt_token_balance
    emit!(DepositEvent {
        user: ctx.accounts.user.key(),
        adapter: crate::ID,
        amount,
        receipt_amount: shares,
        timestamp: clock.unix_timestamp,
    });
    ```

    See `programs/adapter-kamino/src/instructions/deposit.rs` for the full account constraints.
  </Step>

  <Step title="4. Withdraw (45 min)">
    Args: **shares to burn** (receipt tokens), not underlying amount.

    ```rust
    use yield_adapter_trait::user_position_underlying_value;

    let underlying = user_position_underlying_value(
        shares_to_burn,
        vault.total_underlying,
        vault.total_shares,
    )?;

    // Transfer underlying vault → user with vault_authority PDA signer
    emit!(WithdrawEvent { /* amount: underlying, receipt_burned: shares_to_burn */ });
    ```
  </Step>

  <Step title="5. current_value (30 min)">
    <Warning>
    `CurrentValueEvent.value` MUST be the caller's position in **underlying token units**, not the global share price.
    </Warning>

    ```rust
    use yield_adapter_trait::user_position_underlying_value;

    let value = user_position_underlying_value(
        position.receipt_token_balance,
        vault.total_underlying,
        vault.total_shares,
    )?;

    emit!(CurrentValueEvent {
        user: ctx.accounts.user.key(),
        adapter: crate::ID,
        value,
        timestamp: clock.unix_timestamp,
    });
    ```

    `current_value` must not burn shares or transfer tokens.
  </Step>

  <Step title="6. Tests (1–2 h)">
    Copy `tests/helpers/adapter.ts` patterns:

    ```typescript
    await program.methods.deposit(new BN(1_000_000), new BN(0)).accounts({
      user: authority.publicKey,
      vaultState: vaultStatePda,
      userPosition: adapterUserPositionPda(authority.publicKey, program.programId),
      userTokenAccount,
      vaultTokenAccount,
      tokenProgram: TOKEN_PROGRAM_ID,
      systemProgram: SystemProgram.programId,
    }).rpc();
    ```

    Minimum cases:

    - First deposit 1:1 shares
    - Withdraw partial shares
    - `current_value` after deposit (underlying &gt; 0)
    - `deposit(0, 0)` fails (ZeroDepositAmount)
    - `withdraw` over balance fails
    - `deposit(1000, 1001)` fails (SlippageExceeded)
    - `withdraw(1000, 1_000_000_001)` fails (SlippageExceeded)

    Details: [Testing guide](/guides/testing).
  </Step>

  <Step title="7. Register (10 min)">
    Pass the vault state seed and vault authority seed your adapter uses for its PDAs. The dispatcher reads these seeds from the registry to validate adapter accounts at runtime.

    ```typescript
    await registryProgram.methods
      .proposeAdapter(
        "My Adapter",                          // name
        "https://example.com/adapter.json",    // metadata_uri
        "my_vault_state",                      // vault_state_seed
        "my_vault_authority",                  // vault_authority_seed
      )
      .accounts({ /* registryState, adapterEntry, adapterProgram, underlyingMint */ })
      .rpc();
    ```

    Governance (or guardian) calls `approve_adapter`. Until approved, the [dispatcher](/dispatcher) rejects CPI.

    To transfer governance to a multisig, use the two-step pattern:

    ```typescript
    // Step 1: current authority nominates
    await registryProgram.methods
      .nominateGovernance()
      .accounts({ authority, registryState, newAuthority: multisigPda })
      .rpc();

    // Step 2: multisig accepts
    await registryProgram.methods
      .acceptGovernance()
      .accounts({ signer: multisigPda, registryState })
      .rpc();
    ```

    See [Registry](/registry).
  </Step>

  <Step title="8. Optional — protocol CPI (production)">
    After the reference vault works, replace the SPL-only path with CPI into your protocol's deposit/withdraw instructions. Pass `remaining_accounts` for the protocol program; study Kamino's `protocol.rs` in this repo for **attestation-only** fork patterns vs full CPI.

    Protocol-specific errors: codes **7000+**.
  </Step>
</Steps>

## Conformance checklist

<Check>Exactly three instructions: `deposit`, `withdraw`, `current_value`</Check>
<Check>`DepositEvent`, `WithdrawEvent`, `CurrentValueEvent` on every success path</Check>
<Check>`current_value` reports per-user underlying value</Check>
<Check>`AdapterPosition` PDA per user (`ADAPTER_POSITION_SEED` + user pubkey)</Check>
<Check>All math uses `checked_*`</Check>
<Check>`amount > 0` on deposit and withdraw</Check>
<Check>`status.can_deposit()` on deposit, `status.can_withdraw()` on withdraw</Check>
<Check>`shares >= min_shares_out` on deposit, `underlying >= min_underlying_out` on withdraw</Check>
<Check>`user_token_account.mint == vault.underlying_mint`</Check>
<Check>Vault outbound transfers signed by vault authority **PDA**</Check>
<Check>Tests: deposit → current_value → withdraw</Check>
<Check>`vault_state_seed` provided when proposing in registry</Check>
<Check>`cargo clippy --workspace -- -D warnings` clean</Check>

## Common pitfalls

| Pitfall | Fix |
|---------|-----|
| Share price in `CurrentValueEvent` | Use `user_position_underlying_value` on the user's `receipt_token_balance` |
| `AdapterPosition` owned by wrong program | Use `yield_adapter_trait::define_adapter_position!()` in **your** `state.rs` |
| Stale TypeScript client | Run `bash scripts/build-idls.sh` after changing accounts |
| Withdraw by underlying amount | Standard withdraw burns **shares**; compute underlying pro-rata |
| Skipping events | Every deposit/withdraw/value query must emit the standard event |
| No slippage protection | Pass `min_shares_out` / `min_underlying_out` and check after calculation, before any transfers |
